## 🚀 1.20.1 - 7 April 2026

#### ✨ New

- **Opt-in in-memory response cache** — new `cache` option caches `GET` responses in memory with a configurable TTL. Pass `true` for defaults (5-minute TTL, no size limit) or a `CacheOptions` object for a custom budget. Mutations are never cached. Disabled by default — no behaviour change for existing consumers.

```ts
// Default TTL (5 minutes)
const tmdb = new TMDB(token, { cache: true });

// Custom TTL, bounded memory, and per-endpoint exclusions
const tmdb = new TMDB(token, {
	cache: {
		ttl: 60_000,
		max_size: 500,
		excluded_endpoints: ["/trending", /\/discover\//],
	},
});
```

- **`tmdb.cache` controls** — invalidate a single cache entry after a mutation, wipe everything on sign-out, or inspect the current size at runtime:

```ts
tmdb.cache?.invalidate("/movie/now_playing", { language: "en-US" });
tmdb.cache?.clear();
console.log(tmdb.cache?.size);
```

#### 📦 NPM Package

- Added minification option to reduce package size (no changes to the public API)
- Updated README for NPM package

#### 📖 Docs

- New [Response Caching](/docs/getting-started/options/cache) guide covering: enabling with `true`, `CacheOptions` reference, how cache keys work, lazy TTL eviction, `max_size` FIFO eviction, `excluded_endpoints`, the `tmdb.cache` invalidation API, interaction with deduplication, and typical use cases.
- [Features](/docs/getting-started/features) and [Options](/docs/getting-started/options) pages updated with response caching sections.

#### ✨ New

- **[Lists API (v3)](/docs/api-reference/lists)** — full CRUD for TMDB v3 lists via `tmdb.lists`: create, details, update, clear, delete, add items, remove items, and list item details endpoints all supported with full typings and pagination where applicable. - **[Guest Sessions API](/docs/api-reference/guest-sessions)** — read rated content for a guest session via `tmdb.guest_sessions`:
- **Opt-in auto rate limiting** — new `rate_limit` option automatically queues requests to stay within TMDB's API budget using a sliding-window algorithm. Pass `true` for defaults (~40 req / s) or a `RateLimitOptions` object for a custom budget. Applies to both read and mutation requests. Disabled by default — no behaviour change for existing consumers.

```ts
// Default limits (~40 req / s)
const tmdb = new TMDB(token, { rate_limit: true });

// Custom budget
const tmdb = new TMDB(token, { rate_limit: { max_requests: 20, per_ms: 1_000 } });
```

Requests exceeding the budget are held in a FIFO queue and dispatched as a smooth drip — as soon as any slot frees up, the next caller proceeds. No manual batching or `sleep()` calls needed.

## 🚀 1.19.0 - 5 April 2026

#### ✨ New

- **v4 Auth API** — `tmdb.v4.auth` now exposes the full TMDB v4 authentication flow:
  - [`create_request_token()`](/docs/api-reference/v4/auth/create-request-token) — generate a request token for the user to approve (step 1).
  - [`create_access_token()`](/docs/api-reference/v4/auth/create-access-token) — exchange an approved request token for a permanent user access token (step 3).
  - [`delete_access_token()`](/docs/api-reference/v4/auth/delete-access-token) — revoke an access token to log a user out.

```ts
const { request_token } = await tmdb.v4.auth.create_request_token({
	redirect_to: "https://yourapp.com/callback",
});
// redirect user → https://www.themoviedb.org/auth/access?request_token={request_token}
const { access_token, account_id } = await tmdb.v4.auth.create_access_token({ request_token });
```

## 🚀 1.18.1 - 4 April 2026

#### ✨ New

- **Image path autocompletion** — new opt-in `images.autocomplete_paths` option automatically rewrites relative TMDB image paths in API responses to full URLs. Affects `poster_path`, `backdrop_path`, `profile_path`, `logo_path`, `still_path`, and image collection `file_path` fields. Disabled by default to preserve original response semantics.

```ts
const tmdb = new TMDB(token, {
	images: {
		autocomplete_paths: true,
		default_image_sizes: { posters: "original" },
	},
});

const movie = await tmdb.movies.details({ movie_id: 550 });
console.log(movie.poster_path);
// → "https://image.tmdb.org/t/p/original/e1mjopzAS2KNsvpbpahQ1a6SkSn.jpg"
```

## 🚀 1.17.4 - 30 March 2026

#### ✨ New

- **Request interceptors** — hook into the request lifecycle before any TMDB API call is made. Pass a single [`RequestInterceptor`](/docs/types/options/interceptors#requestinterceptor) or an array; each interceptor receives a [`RequestInterceptorContext`](/docs/types/options/interceptors#requestinterceptorcontext) (`endpoint`, `params`, `method`) and may return a modified context to override the endpoint or params, or `void` to leave the request unchanged. Async interceptors are fully supported and awaited in order.
- **Response interceptors** — two new hooks under `interceptors.response`:
  - `onSuccess` — called after every successful response, before data is returned to the caller. Designed for side effects such as logging, caching, or image URL enrichment. Must not change the shape of the returned data.
  - `onError` — called after a [`TMDBError`](/docs/types/options/interceptors#responseerrorinterceptor) is created. Receives a fully-typed `TMDBError` for side effects (Sentry, toast notifications, etc.). The error is always re-thrown after the interceptor runs.

```ts
const tmdb = new TMDB(token, {
	interceptors: {
		request: [
			(ctx) => console.log(`→ ${ctx.method} ${ctx.endpoint}`),
			(ctx) => ({ ...ctx, params: { ...ctx.params, include_adult: false } }),
		],
		response: {
			onSuccess: (data) => myCache.set(data),
			onError: (error) => Sentry.captureException(error),
		},
	},
});
```

#### 📖 Docs

- New [Interceptors](/docs/getting-started/options/interceptors) guide covering request and response hooks with full examples.
- New [Interceptors types](/docs/types/options/interceptors) page documenting `RequestInterceptorContext`, `RequestInterceptor`, `ResponseSuccessInterceptor`, and `ResponseErrorInterceptor`.

## 🚀 1.17.3 - 29 March 2026

#### 🐞 Bug Fixes

- **Path params leaking into query strings** - path parameters (e.g. `movie_id`, `season_number`, etc.) were mistakenly being included in the query string of requests instead of being injected into the URL path. This has been fixed by centralizing all parameter serialization logic in the `ApiClient`, ensuring that path params are correctly placed in the URL and not duplicated in the query string.
- **Malformed route in `tv_episodes.changes()`** — the generated path contained double slashes (`/tv//episode/{id}//changes`). Now correctly builds `/tv/episode/{episode_id}/changes`.
- **`| null` removed from image path types** — `poster_path`, `backdrop_path`, `still_path`, `logo_path`, `profile_path`, and `runtime` fields no longer include `| null` in their types. The client's `sanitizeNulls` already converts `null` → `undefined` at runtime, so the static types now match the actual runtime behaviour.

#### ✨ New

- **Image path type guards** — five new exported helpers for narrowing image-path fields on unknown objects: [`hasPosterPath`](/docs/types/utility#hasposterpath), [`hasBackdropPath`](/docs/types/utility#hasbackdroppath), [`hasProfilePath`](/docs/types/utility#hasprofilepath), [`hasStillPath`](/docs/types/utility#hasstillpath), [`hasLogoPath`](/docs/types/utility#haslogopath).
- New [Utility](/docs/types/utility) types page documenting the type guards and other utility types exported by the package.

#### 🧪 Tests

- Closed remaining coverage gaps across the SDK; added regression tests for array-valued params to guard deduplication key stability and query serialization alignment.
